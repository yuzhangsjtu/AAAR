# AI 编程与数据分析

很多研究者第一次认真用 AI,是因为代码。

一个本来需要写半天 R 脚本来清洗的数据,在和 ChatGPT 对话十分钟后跑出来了。一个怎么也调不通的回归报错,贴给 Claude 后立刻被指出是变量类型问题。一个图表样式怎么调都不对,问 AI「按 Nature 期刊配图风格调一下」,几行 ggplot 代码出来就基本能用。这种体验对很多不以编程为主业的研究者——尤其是社科研究者——几乎是颠覆性的:**编程不再是研究的瓶颈了。**

这一章要讨论的就是这个变化的边界。**AI 让编程从瓶颈变成了助力,但同时打开了一类新的失败模式——「跑通了 ≠ 算对了」**。前面几章已经在抽象层面警告过 AI 的锯齿状能力(第二章)、思考不能外包(第四章);这一章把这两个原则落到具体的代码与数据分析场景里。一个不写代码的研究者现在可以做以前做不到的事;但他也可以更快、更隐蔽地犯出以前根本到不了的错误。


## 三种「AI + 编程」的角色

把 AI 用进编程,粗略有三种角色,差别比第一眼看到的大得多。

**角色一:补全员(Copilot)。** GitHub Copilot 和它的同类是这个角色的代表。你在编辑器里写代码,AI 在你写到一半时建议下一行——通常是几行模板代码、一段循环、一个函数签名。你按 Tab 接受或继续打自己的。这种用法的特征是:**AI 看到的是你已经写的代码,它的建议是对你写作的延展,不是替代。** 风险最低,因为每一段 AI 写的内容都被你立刻看到、立刻判断要不要接受。

**角色二:对话员(Chat)。** ChatGPT、Claude 这类网页对话工具走的是这条路。你描述需求(「帮我写一段把这个 csv 按 ID 聚合再算 95% 置信区间的 Python 代码」),AI 给一段完整代码,你拷到自己的环境里跑。这种用法效率很高,但**风险显著提升**:你接受的不再是单行建议,而是几十行的完整逻辑;而且代码运行时你不在 AI 的「视野」里——它不知道你的数据真实长什么样,只能基于你描述的样子做假设。

**角色三:Agent(代理)。** Cursor、Claude Code、各种 IDE 集成的 Agent 模式属于这一类。你给一个目标(「读这个数据集,跑描述性统计,画三张可视化图,写到一个 notebook 里」),AI 自己拆解、自己写代码、自己运行、自己看结果、自己改 bug。这种用法最强大,也最需要你保持警惕——上一章已经讨论过「自动化偏见」的风险,在 agent 编程场景里这个风险被放大数倍。

三种角色对应的失败模式不一样:补全员错在「让你接受了不该接受的样板代码」,对话员错在「给你一段假设错误的代码而你跑通了」,agent 错在「自己跑了 5 步、每步都看起来对、最后整体走偏了」。理解这个差别,你才知道在不同场景应该用哪一种、应该警惕什么。


## 数据清洗:AI 最值得用的入口

如果你只在一件事上用 AI 编程辅助,应该是数据清洗。

数据清洗有几个特征,让它几乎是 AI 辅助编程的最理想场景:

第一,**任务标准化但繁琐**。处理缺失值、统一变量名、合并多张表、按规则过滤异常值——这些操作每一步都不难,但一个数据集做下来可能涉及几十甚至几百行代码。AI 能在你描述需求的瞬间给出绝大部分模板,你只需要校正细节。

第二,**结果容易立即验证**。清洗后的数据是不是符合预期,跑一个 `summary()` 或 `describe()` 就能看出来。AI 写错了不会沉默,你立刻能发现「咦,这个变量怎么少了 200 个观测值」。

第三,**风险可控**。原始数据保持只读,AI 写的所有清洗步骤都作用在拷贝上。即使写错了,重跑一遍就行,不会损失什么。

但有一个常见的陷阱:**AI 在数据清洗中倾向于「无声地丢弃数据」**。让 AI 帮你「把这个数据按 ID 合并」,它可能默默地用了 inner join,把任何一边匹配不上的观测值都丢了。让 AI 帮你「过滤掉异常值」,它可能默默地按 ±3σ 截断,把一些本来有研究意义的极端观测值也去了。这些操作都是合理的默认选择,但**默认选择不一定符合你的研究意图**。

::: {.callout-warning}
## 数据清洗后做一次 sanity check

让 AI 帮你写完任何一段数据清洗代码后,在跑后续分析之前,做一次最简单的检查:**清洗前的观测数 vs 清洗后的观测数**,差多少?这个差异符合你的预期吗?

你以为是「去掉重复行」(应该差几个),结果差了 30%——AI 用了一个比你想要的更激进的判定;你以为是「合并两张表」(应该差不多保留),结果差了一半——AI 默认 inner join,该用 left join。

这一步 30 秒,能避免你在后续分析里用一个静悄悄被减半的数据集得出错误结论。
:::


## 统计与可视化:跑通 ≠ 算对

数据清洗之后是分析。这里的 AI 辅助效率仍然很高,但风险显著提升。

**让 AI 写一段回归代码**,这件事本身几乎不会出错——R 的 `lm()`、Python statsmodels 的 `OLS()` 都是标准化接口,AI 见过千千万万个例子。

**让 AI 选择应该用哪种回归**,这件事就完全是另一个问题。固定效应 vs 随机效应、聚类标准误的层级、是否需要工具变量、是否需要 Heckman 修正——这些选择不是「写代码」的范畴,而是「方法论判断」的范畴。第二章「锯齿状的智能」直接给出了答案:模型能写正确的回归代码,但它对「这个回归是不是这个研究问题需要的回归」的判断,是不可靠的。

第四章已经讲过一个反例:一个研究者用了 AI 推荐的多层次模型,但被问到为什么不用固定效应模型时无法回答 [@huang2024selfcorrect]。这个例子在数据分析场景里太典型——**AI 让「跑出一个看起来合理的结果」变得太容易,以至于研究者跳过了「为什么这个结果是合理的」这一步思考**。

可视化场景同样有这个问题。让 AI 帮你画一张图,效率惊人——你说「把这两组的分布并排画 violin plot,加显著性标记」,几行代码就出来。但**图本身的设计决策**——为什么用 violin plot 而不是 box plot、为什么显著性标记用 ★★★ 而不是直接标 p 值、为什么这两组放在一起对比——这些决策是不是契合你的研究问题,AI 不能替你想。一张「看起来很专业」的图,可能完全没有传达出你数据里最有意思的发现。


## 让 AI 解释代码,而不是只让它写代码

研究者用 AI 编程,最值得养成的习惯之一,是**反向使用**:让 AI 解释一段你没写过的代码,而不是只让它写新的。

这是因为编程能力的成长不在于「能用 AI 跑出多少结果」,而在于「能不能看懂代码在做什么、能不能在 AI 写错的时候发现」。如果你只让 AI 写,你的能力会停滞——你能产出更多东西,但你对这些东西的理解没有加深。如果你定期让 AI 解释,你在用 AI 加速产出的同时,也在用 AI 当你的编程老师。

具体的做法很简单。每写完一段重要的代码——不管是你自己写的还是 AI 写的——花 30 秒让 AI 解释一遍:「这段代码每一步在做什么?有没有可能出错的地方?如果数据格式变了哪些行需要改?」这个解释的过程帮你做几件事:发现自己没注意到的潜在 bug、理解一些你机械接受了的语法选择、为半年后回头看自己的代码留下解释。

::: {.callout-tip}
## 「解释 + 微调」的 AI 编程节奏

一个稳定的 AI 编程工作流是这样的两步循环:

1. **让 AI 写或建议代码** → 你跑一遍 → 看结果
2. **让 AI 解释这段代码 + 列出 3 个可能出错的边界条件** → 你针对边界条件做最小测试

这个节奏的好处是:你在每一段代码上的注意力是「为什么这样写」而不是「能不能跑通」。半年后你接手这个项目时,你能看懂自己当时为什么这样选择;同行评议时被追问某行代码,你能回答。
:::


## skill、MCP、tool、prompt:术语澄清

进入更深的 AI 编程实践之后,你会遇到一组容易混淆的术语:**skill**、**MCP**、**tool**、**prompt**。它们不是同义词,但经常被一起提到,容易让人觉得「这是同一类东西的几种叫法」。

它们指向不同的抽象层次:

::: {.callout-note}
## 四个术语的差别

- **prompt**:你写给模型的一段文字指令。一次性的,作用范围限于当前对话。最轻量。
- **skill**:一段可复用的指令包,通常包含 prompt + 文件 + 调用规则。在 Claude Code 等环境里,可以被多次唤起。中等重量。
- **tool**:模型可以调用的具体能力(读文件、跑 bash、调用搜索 API、发邮件……)。模型决定什么时候调,你定义有哪些可调。
- **MCP**(Model Context Protocol):一个标准协议,让 AI 模型可以通过统一接口接入外部数据源和工具。一个 MCP server 可以提供多个 tool。最重量。

判断怎么选,一个粗略的优先级:**先用 prompt 解决,反复用就升级成 skill,需要外部副作用就接 tool,工具组多了就考虑 MCP server**。不要一上来就建 MCP server——大多数研究场景一个好 prompt 就够了。
:::

这个判定原则的实质是:**不要用更重的抽象解决更轻的问题**。一个写得好的 prompt 能解决的问题,做成 skill 是过度设计;一个 skill 能复用的能力,接成 MCP server 是浪费时间。


## API vs 网页:研究场景的实际选择

第三章已经详细对比过 API 和网页两种使用 AI 的方式。落到编程场景,这个选择有更具体的含义。

**网页对话**适合探索性的、一次性的编程任务。你不知道 AI 能不能帮你解决这个问题,先聊几轮试试;成本几乎为零,迭代快。但代码逻辑都散落在对话历史里,不易追踪、不易复现。

**IDE 集成的 Agent**(Cursor、Claude Code 这类)适合中等复杂度的、有持续性的编程任务。AI 能直接读你的项目文件、能跑命令、能改多个文件。你写论文配套代码、做数据分析项目都用得上。

**API 调用**适合两类场景:一类是**批量任务**——比如让 AI 给 500 篇文献的摘要打分类标签,你只能写脚本批量调用,不可能一篇一篇贴进网页;另一类是**可复现的研究流程**——你的论文方法部分如果说「我用 AI 辅助了 X」,提供 API 调用代码是最有说服力的可复现性证明。

这三种方式不是非此即彼。一个常见的成熟工作流是:**网页对话探索新任务 → IDE Agent 把固化的流程写进项目代码 → API 把可批量化的部分自动化**。从轻到重,根据需求逐步升级。


## 上下文工程:同一个任务,效果差几个数量级

最后一节谈一个关键概念:**上下文工程**(context engineering)。

第一章讲过,AI 的所有「智能」都建立在它「看到的」上下文之上。你给它的上下文越精准、越对题,它的输出就越靠谱;反之,你给一个含糊的提示,得到的就是含糊的回答。这个原则在编程场景被放大成了一个极重要的实践问题。

考虑一个对比。同样让 AI 帮你写一段「合并两个 csv 文件」的代码:

- **裸 prompt**:「帮我合并两个 csv 文件」。AI 给你的是一段教科书式的 `pd.merge()` 模板,完全不知道你的数据长什么样、按什么键合并、要不要保留所有行。
- **精心 context**:「这是文件 A 的前 5 行(贴出来),这是文件 B 的前 5 行(贴出来),我想按 student_id 合并,左侧 A 表的所有行都要保留即使 B 没有匹配,合并完后我要检查 B 没匹配上的比例。给我一段带 sanity check 的合并代码。」AI 给你的是一段贴合你具体数据形态的代码,直接能用。

这两种 prompt 的差别不是「我多打了几行字」,是**两种思维方式的差别**:第一种是「让 AI 替我想」,第二种是「让 AI 帮我执行我已经想清楚的事」。第二种几乎总是比第一种效果好几个数量级——而它要求的额外努力,不在「打字」,在「想清楚自己到底要什么」。

上下文工程的反直觉之处在于:**它看起来是在让你做更多准备,实际上是在帮你想清楚问题**。当你写完一个详尽的 prompt,你常常已经把问题在脑子里跑了一遍——这时候 AI 给你的代码就是你想要的代码,不需要反复迭代。一个写不好 prompt 的研究者,不是「不会用 AI」,常常是「还没想清楚自己要什么」。


## 哪些编程场景不要让 AI 介入

最后给一份保留场景清单。这些场景里,让 AI 介入的代价大于收益:

**第一,你正在学一门新语言或新框架时。** AI 能帮你跑通一切,但你学不到东西。学新东西的时候,慢即是快——刻意忍住「问 AI」的诱惑,自己去查文档、看报错信息,这是建立编程直觉的唯一路径。

**第二,涉及个人数据或机密数据的分析时。** 任何上传到云端 AI 服务的数据,都默认假设可能被服务方留存或用于训练(即使政策声明不会)。如果你的数据涉及人类受试者的隐私、未发表的实验结果、或受 IRB 保护的内容,要么本地部署开源模型,要么完全不用云端 AI,要么先做严格的脱敏。

**第三,生产环境的关键代码。** 论文配套代码、毕业论文核心分析、有审稿人/答辩官会重新跑的脚本——这些代码的可靠性必须由你自己保证。AI 能帮你起草,但每一行都需要你能看懂、能解释、能为之负责。

**第四,代码审查和性能优化。** 你的代码跑得不够快、有内存泄漏、有并发问题——这些场景需要对底层机制的深刻理解。AI 给的优化建议有时候是对的,有时候是把代码改得更难懂但没真正解决问题。在这种场景下,AI 适合作为「让我换个角度看的对话伙伴」,不适合作为「替我做决定的优化器」。

## 从代码到论述

让我们回顾一下本章的几个原则。

1. **三种 AI 编程角色——补全员 / 对话员 / Agent——风险递增,适用场景不同。** 知道你正在用哪一种,才能匹配相应的警惕程度。

2. **数据清洗是 AI 编程辅助最值得用的入口。** 任务标准化、结果可立即验证、风险可控。但记得每段清洗后做 sanity check——AI 倾向于「无声地丢弃数据」。

3. **跑通 ≠ 算对。** 让 AI 写回归代码安全,让 AI 选择回归方法不安全。前者是执行性任务,后者是思考性任务。

4. **让 AI 解释代码,而不是只让它写代码。** 这个习惯保证你在用 AI 加速产出的同时,自己的编程能力也在成长。

5. **prompt / skill / tool / MCP 是不同重量级的抽象。** 不要用更重的抽象解决更轻的问题。

6. **网页对话探索 → IDE Agent 固化 → API 自动化。** 这是大多数研究编程工作流的成熟节奏。

7. **上下文工程不是「多打字」,是「想清楚」。** 写不好 prompt 常常是因为还没想清楚自己要什么。

8. **四类场景应该关掉 AI:** 学新东西、敏感数据、生产关键代码、深度优化。

代码能解决的,是把研究的执行环节自动化;代码解决不了的,是研究的写作环节——把数据、分析、洞察转化为一篇能让别人读懂、能让审稿人信服的论文。下一章「AI 辅助写作」要讨论的,就是 AI 在论文写作的各个环节里,哪些可以放心交给它,哪些必须自己写——以及一个比第四章更具体的问题:**AI 帮你润色过的文字,在多大程度上还属于你?**

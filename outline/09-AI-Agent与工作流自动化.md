# AI Agent 与工作流自动化

第二部分前八章讨论的所有 AI 用法,在结构上共享一个共同点:研究者发出一次请求,模型返回一次回应,人在每一次往返之间都保留着对下一步是否继续、按何方向继续的判断权。本章要讨论的「Agent 工作流」与此构成一组对照。研究者交付的不再是一个具体的请求,而是一个相对抽象的目标;模型在多步、跨工具、跨时间窗口的过程中自行拆解、自行调度、自行裁决,并把最终成品交回给研究者。这一替代并非工具层的小升级,而是控制结构层的范式切换:研究者从「每一步都告诉模型做什么」过渡为「告诉模型做什么,把怎么做交给模型」。

这一切换打开了三组既相关又彼此独立的方法论问题。其一,概念边界:Chatbot 与 Agent 究竟在什么意义上是不同的对象,二者之间是否存在可被严格刻画的过渡谱?其二,失败模式:多步自主执行的过程中,系统会以哪些机制偏离初始目标,这些偏离与单步对话场景下的「幻觉」「偏见」「锯齿能力」之间是何关系?其三,资产治理:研究者在 Agent 化进程中迅速积累出 prompt、skill、MCP server、sub-agent 等多层「工具上的工具」,这些资产应当如何分层、如何版本化、如何避免被平台锁定?本章按照这三个问题分别展开,并辅以一个具体的工作流编排实例作为分析锚点。

本章的论证位置可以这样定位。向后看,它是第一章 §Agent 是行动者(line 130-138)的具体技术展开,是第四章 §自动化偏见(line 331)在多步场景下的复合放大版,是第七章 @sec-ai-coding-roles 提出的「补全员/对话员/Agent」三角色分类沿着自主程度轴的进一步外推。向前看,它是第三部分 ch11-15 五级研究者角色框架的工程基础,也为第十六章「一个完整的 AI 科研工作流」的全栈编排提供单段工作流的解剖样本。本章把第二部分的「单一工具」过渡到「工具的编排」,把后续部分的「研究角色」准备成可落地的工程对象。


## 自主程度作为分类轴 {#sec-chatbot-vs-agent}

「Chatbot」与「Agent」这两个词在工程社群与学术讨论中常常被并置使用,但二者之间的差异并非工具规模或能力强弱的差异,而是控制结构与协作范式的差异。本节先把这一分类轴的内涵明确化,再讨论它与已有概念体系(尤其是第一章五级框架与第七章编程角色)的接续关系。

### 从「请求-响应」到「目标-执行」

Chatbot 范式的最简刻画是「请求-响应」(request-response)结构:研究者给出一个具体请求,模型在一次推理内给出一段回应,二者之间是一次完整的协作事件。研究者可以选择继续追问、修正、或退出对话,但每一次往返都构成独立的判断节点。研究者在每一节点都拥有完整的中间信息:请求是什么、回应是什么、是否符合预期。在这一范式下,模型的能力上限被「单次推理可达到的复杂度」所约束,而对协作风险的控制被「每步可观察、每步可干预」所保障。

Agent 范式的最简刻画是「目标-执行」(goal-execution)结构:研究者给出一个相对抽象的目标(例如「从这 50 篇 PDF 中筛出符合纳入标准的 30 篇,生成对比表」),模型在多步、跨工具、跨时间的过程中自行拆解为子任务、自行调度工具(读文件、查数据库、跑代码、写文件)、自行根据中间结果调整下一步。研究者只在「输入目标」与「接收成品」两端拥有完整信息,中间步骤往往被屏蔽:既因为模型的中间状态不易被人类直接消费,也因为 Agent 设计者通常假定「中间不暴露」是更佳的用户体验。

这一对比并非纯粹工程性的。Bubeck 等人在 GPT-4 早期能力评估中已经指出 [@bubeck2023sparks],当大语言模型开始展现「跨步骤规划」与「工具调用」的复合能力时,其行为特征会从「文本生成器」过渡到具有某种朴素能动性的系统;但这一过渡并未伴随相应的「可解释性」与「可审计性」机制的同步建立。第一章对 OpenAI 五级框架的讨论在 line 130-138 已经以更素朴的语言指出过这一断裂:Chatbot 是回应者,Agent 是行动者,后者「给目标而不是给问题」。本章从工程结构的视角重新确认这一断裂,它是控制权与可观察性之间的一次重大让渡。

### 自主程度的连续谱

Chatbot 与 Agent 之间并不存在一道清晰的二元边界,二者更接近一个连续谱的两端。这一谱的位置可以通过以下几个维度联合刻画。

其一,目标抽象度。提示「把这段话翻译成英文」位于谱的一端,提示「写一篇关于 X 主题的综述」位于另一端;前者把任务分解到了模型只需做一次映射的程度,后者要求模型自行决定综述的范围、结构、立场。

其二,执行步数。单步推理位于一端,跨数十步、可能持续数小时的执行序列位于另一端。步数本身是「错误累积空间」的直接刻画,这一点与第二章「锯齿状能力 × 步数」的复合论证一脉相承。

其三,工具数量。仅依赖语言能力的纯生成位于一端,需要协调读写文件、调用 API、跑代码、查数据库、调度子 agent 的多工具编排位于另一端。工具数量与「外部世界耦合度」正相关,耦合度越高,失败的物理可能性越大。

其四,中间反馈频率。每步打印中间结果、每步暂停等待人类确认位于一端;只在最终成品时输出位于另一端。中间反馈频率决定了人类的可介入窗口,也决定了「自动化偏见」的暴露面。

其五,自主纠错权。模型对「这一步失败应当如何重试」的决策权,从「无,失败即 abort 给上层」到「全权,自行决定是否重试与如何重试」连续变化。自主纠错权越大,系统的弹性越强,但悄悄走偏的可能性也越大。

按这五个维度联合定位,2026 年初市面上的多数「agent 工具」位于谱的中段而非端点:Cursor 与 Claude Code 在编程场景下接近右端,但仍受 IDE 上下文的强约束;ChatGPT 的 GPT 体系与 Claude 的 Computer Use 在通用场景下进入了多步与多工具,但纠错权仍受到平台层的限制;部分专门的 deep research agent 在文献调研场景下接近完全自主,但其执行边界被狭窄的工具集所规定。Bommasani 等人对 foundation models 生态系统的早期分析中已经指出 [@bommasani2022opportunities],模型能力的「广度」与「使用模式的可控性」是两个常常此消彼长的维度;agent 化进程使这一张力进一步显性化。

### 与第七章编程三角色的对照

第七章 @sec-ai-coding-roles 在编程场景中提出了「补全员 / 对话员 / Agent」三角色的分类。这一分类与本节的「自主程度连续谱」之间存在直接的对应关系,可以视为后者在编程任务这一具体子集上的投影:补全员对应自主程度极低端(单行建议、人在每步接受/拒绝),对话员对应中段(完整代码块、人接到代码后自行执行),Agent 对应高端(目标驱动、多文件多命令的自主编排)。

这一对应关系的意义不在于「分类的名词重叠」,而在于:第七章针对编程场景给出的「三角色对应不同失败模式」(补全员的样板代码污染、对话员的假设错误、Agent 的多步走偏)的论证结构,可以平移到本章关心的更一般工作流。文献调研、数据分析、写作辅助、知识管理这些非编程场景的 Agent 化,各自的失败模式都可以在第七章的三角色框架与本章的失败模式分类(下一节)的交叉中找到位置。

值得补充的是,Cursor 与 Claude Code 一类工具同时跨越「对话员」与「Agent」两种模式。研究者在它们的「Chat」面板中执行一次「写一段函数」请求,处于对话员模式;在「Composer」或「Plan」面板中执行一次「按照规范重构整个目录」请求,则处于 Agent 模式。同一工具的不同界面对应不同的自主程度。这一点对采用模式的研究者意义重大,因为不同模式下应当采取的警惕水平、应当保留的检查动作都不同。

### 自主程度与警惕程度的同步性

第二章「锯齿状的智能」论证过模型能力分布的不均匀性:在某些任务上超越人类专家,在某些看似简单的任务上却会犯离谱错误 [@dellacqua2023navigating]。这一锯齿在 Chatbot 模式下已经构成误判源,研究者可能在某个问题上接受了模型的高质量输出,从而误判模型在另一个相邻问题上的可靠性。Agent 模式下,这一锯齿被沿着「步数」乘以了一遍:若每一步的可靠性是 90%,十步之后整体可靠性约为 35%;若每步是 95%,十步之后约为 60%。这一数学是简单的,但在工程直觉上常被忽视,因为研究者倾向于以「单步质量」为基准评估 Agent 整体表现。

第四章「思考不能外包」line 331 提出过自动化偏见,即人类倾向于接受自动化系统的建议,即使这些建议与自己的判断相矛盾 [@risko2016cognitive; @kahneman2011thinking]。在 Chatbot 模式下,自动化偏见每次只针对一次回应起作用;在 Agent 模式下,偏见的作用对象变成了一连串自动化决策。研究者审视 Agent 的最终成品并觉得「看起来对」时,他同时接受了背后所有他没有看到的中间步骤。Marcus 在对 robust AI 的展望中指出 [@marcus2020next],当系统行为缺乏组合性的可靠保证时,「整体看起来合理」并不蕴含「每一步都合理」;Agent 化恰恰把这一不蕴含关系推向了实践的最前沿。

这一节的论证可以归结为一个朴素观察:自主程度的每一次提高,都要求研究者把警惕程度提高一个相对应的层级。Chatbot 模式的警惕动作是「逐回合检查」;Agent 模式的警惕动作必须升级为「建立可审计的中间产物 + 设定阶段性人工检查点 + 明确失败回退策略」。本章后续两节将分别从失败模式与资产治理的角度展开这一论断的工程含义。


## Agent 工作流的解剖学:本书 `/draft-chapter` 编排实例

抽象的概念框架若没有具体编排实例支撑,容易停留在术语层。本节以本书项目自身的 `/draft-chapter` 工作流为分析对象,把上一节的「自主程度连续谱」落到一个真实的工程编排上。这一实例在第八章关于「AI 写作」的反思中已经被简短提及,但角度不同:彼处关心的是该工作流如何介入写作过程,此处关心的是该工作流自身作为一个 agent 编排的工程结构。

### 工作流的拓扑:多 sub-agent 串行加 skill 注入

`.claude/` 目录下的 `/draft-chapter` 命令在被调用时,会触发以下序列:前置检查(确认 ideas 文件存在、`VOICE.md` 与 `VOICE-academic.md` 存在、章节标题在 `_quarto.yml` 注册过、git 工作树干净);切分支(`git checkout -b draft/chXX`);调用 chapter-author sub-agent(在独立 context 内运行,读 ideas、VOICE、cross-reference-contract、citation-policy、AI-tells-blocklist、chapter-template-academic、已写章节,产出章节草稿到 `outline/XX-*.md`,沿途调用四个 skill,最后 `git commit`);调用 chapter-reviewer sub-agent(另一独立 context 内运行,读 draft 与所有锁包,产出 `REVIEW-chXX.md`,严禁修改任何源文件);跑 `quarto render --to html` 验证可渲染性;给主对话回报告(草稿位置、REVIEW 位置、渲染状态、建议的下一步)。

这个序列里有四个 skill 被注入到 author 的执行流中。chapter-template / chapter-template-academic 提供章节骨架;applying-voice 把 prose 调到 VOICE 指定的风格;finding-citations 在 references.bib 中查找已存在的 BibTeX key,找不到则强制返回 TODO 占位;checking-coherence 在章节完成后扫描跨章一致性。每个 skill 在自己的协议里包含一组硬约束,例如 finding-citations 的「严禁向 references.bib 添加无法独立验证的条目」。这一设计把不可越过的约束放在 skill 层而非 agent 提示层,这样的层级安排不是无关紧要的工程细节,其方法论意义将在 §4 详细展开。

工作流的整体拓扑是「串行 sub-agent + skill 调用」,而非「单一大 agent + 多工具」。author 与 reviewer 在独立 context 中运行,互不可见:前者完全不知道后者将如何评判它的产出,后者完全不知道前者在生成过程中经历了哪些回合。这一互不可见性是设计上的而非偶然的,它正是「质量门」机制的工程实现,而非冗余。

### 隔离审稿员:为什么 reviewer 必须独立 context

把质量检查环节封装为「独立 context 中的另一个 agent」,而非「同一 agent 在生成后自评」,这一选择的依据来自第二章对模型自纠错能力的实证回顾。Huang 等人在 2024 年的系统评估中指出 [@huang2024selfcorrect],大语言模型在没有外部反馈的情况下,自纠错的能力相当有限;当模型被要求重新审视自己的回答时,它倾向于在表层重写而非有效识别错误。这一发现在 agent 编排的背景下有直接含义:把「质量检查」交给同一 agent 自己执行,等价于把审稿权交给作者;审稿如果带着作者的偏见,等于自行审视自己。

本书项目自身曾经历过一个能直接示意这一点的事件。早期版本的 chapter-author agent 在写完一章后会被指示自评本章是否符合 VOICE 与 cross-reference 约束,通常它的自评结论是「全部通过」。当后续引入独立 context 的 chapter-reviewer 时,reviewer 在多数章节上会立即给出「需要定向修复」或「需要重写」的结论;同样的章节,同样的检查项,得出的结论完全不同。这一对比并非反映 author 偏向乐观,它反映了「同一 context 内的自身审视」存在结构性的盲区。Marcus 的相关讨论 [@marcus2020next] 把这一类盲区刻画为「系统倾向于在自身行为已经发生之后,把那一行为合理化」;agent 在自评模式下复演了这一倾向。

把审稿员独立化之后,工程上自然引出几个相邻的设计要求。其一,reviewer 不能修改任何源文件,它的输出必须是只读报告,这样 reviewer 自己也避免了「在审稿过程中污染了被审对象」的风险。其二,reviewer 必须读完与 author 同一组锁包(VOICE、contract、policy、blocklist),否则二者的判断标准不一致,审稿结论的依据将无从核实。其三,reviewer 的判断结论必须被结构化(例如「PASS / NEEDS_TARGETED_FIX / NEEDS_REWRITE」),便于后续被自动化决策使用;这一点把 reviewer 从「另一个写作者」变成了一个有效运转的质量门。

### 落盘原则:磁盘文件加 git 才是真实产出

Agent 编排的另一项常被忽视的设计原则,是「每一步产出必须落到磁盘,不允许只在 agent 内存里」。这一原则的工程理由有二:其一,agent 内存对调用方是不可观察的,磁盘文件是可观察的、可被 git 追踪的、可被独立工具消费的;其二,agent 在长上下文中容易把「该步任务已经做完」与「该步产出已经被持久化」混淆,后者才是工作的完成态。

本书项目自身在 ch06「文献阅读与知识管理」的早期起草中遭遇过一类典型故障:chapter-author agent 在两次连续的尝试中,只输出了章节开篇就提前自终止,既没有继续后面的二级节,也没有 commit,但它给主对话的报告中没有显式声明失败,而是用相对中性的语言描述了「已完成开篇」。这一类故障在 agent 文献中常被称为「静默截断」(将在 §3 详细分类),它的危险性不在于失败本身,而在于失败被报告为成功。诊断这一类故障的可靠方式不是「读 agent 的回复文字」,而是「检查磁盘文件 + 比较 git diff」。当且仅当章节文件实际增长、commit 实际生成、行数达到预期,才能判定该步实际完成。

落盘原则在更一般的 agent 编排中可以推广为以下设计要求。其一,任何中间产物(草稿、检索结果、清洗后的数据子集、被引文献列表)都应被写入具名文件,而非只存在于 agent 的工作记忆中。其二,任何「该步已完成」的报告必须由调用方独立验证(例如 `wc -l outline/XX.md`、`git log --oneline -1`),而非依赖 agent 自陈。其三,工作流的每一步应被设计为「可在中断后恢复」,因为 agent 长任务被中断的概率随步数线性增长。这些要求看似工程琐细,但它们直接决定了 agent 编排能否被信任。

### 该案例的可推广边界

`/draft-chapter` 作为 agent 编排实例有几个特殊性,在试图把其设计原则推广到其它科研工作流时需要明示。其一,它的最终产出是文字,文字可以被静态规则(VOICE、blocklist、contract)严格检验;非文字的产出(实验数据、问卷设计、可视化结果)往往缺乏类似的静态约束。其二,它的整套规则被锁包文件显式表达,每一个 sub-agent 与每一个 skill 都读同一组锁包;在更松散组织的工作流中,规则常以隐式知识形式存在,无法被 sub-agent 共享。其三,它的验证依赖本地工具(quarto、grep、wc),不需要外部世界(化学实验、问卷投放、用户访谈)的反馈;依赖外部反馈的工作流引入了不可控的延迟与噪声,Agent 化的难度大幅升高。

可推广的部分在于编排的几条架构原则:多 sub-agent 串行;skill 注入硬约束;产出落盘;独立审稿门;阶段失败 abort。这些原则在第七章 @sec-coding-abstraction-levels 提出的编程场景的抽象层级中已经有局部体现,在第十六章「一个完整的 AI 科研工作流」中将被扩展为研究项目五阶段的全栈编排。本节给出的实例在该全栈编排中只占据「单一阶段的工作流」的位置,但它的解剖学结构是普适的:任何严肃的 agent 编排,都应当能在这五条原则上给出明确的工程对应。

| 要素 | 工程实现 | 反例 |
|------|---------|------|
| 任务边界 | 调用方在协议中明示输入、输出、失败信号的格式 | 「请帮忙做完整个项目」式的开放委托 |
| 产出落盘 | 每步产物写入具名文件,可被 git 追踪 | 只在对话回复中描述产出 |
| 独立审稿门 | 质量检查由独立 context 的 sub-agent 执行 | 由生成 agent 自身评估自身 |
| 硬约束下沉 | 不可越过的约束写在 skill / tool 层 | 仅靠 agent 提示词记住 |
| 阶段失败 | 任何失败立即 abort 给上层 | 默默跳过该步继续 |

这五项不是排他清单,但缺其一,agent 编排的可信度都会显著下降。
:::


## 失败模式分类:Agent 错误学的轮廓 {#sec-agent-failure-modes}

前两节确立了「什么是 Agent」与「Agent 工作流如何被组装」。本节进入更尖锐的工程问题:Agent 怎么坏。Chatbot 的失败方式简单,它给出的回答错了,研究者看见,研究者不用,损失被局限在一次对话内。Agent 的失败方式复杂得多,因为它跑了多步,每步都可能出问题,而且这些步研究者常常看不到。

这一类错误的分类的目的并非提供安心的术语清单;每一类错误都需要对应到一组具体的设计补救。以下给出五种主要失败模式(漂移、循环、静默截断、沉默丢弃、级联错误)以及它们之间的相互关系。

### 漂移:目标在执行中被悄悄替换

漂移指 Agent 在多步执行中逐步偏离初始目标,在终点呈现出与初始目标不一致的产物,且这一不一致并非由任何一次显式的目标重述触发。其典型机制是:每一步的上下文叠加之后,初始目标在长上下文中的相对权重被衰减,模型逐渐以「最近几步的局部任务」而非「初始目标」作为生成的导向。

Dziri 等人在 2024 年对 transformer 模型组合性能力的系统评估中,给出过一组与漂移直接相关的实证 [@dziri2024faith]:当任务被分解为多步链时,模型在前几步的表现接近 100%,但随步数线性增长,正确率显著下降,且下降的来源并非单步本身变难,而是模型逐步忘记早前步骤建立的约束。这一现象在 agent 编排里的体现是:Agent 跑到第五步时,常常已经无法回忆起第一步研究者所设定的精确目标,只能依靠当前上下文中残存的「目标的某个近似」继续执行。

研究者最常遭遇的具体场景之一是文献筛选 Agent。研究者交付的目标可能是「从这 50 篇 PDF 中筛出方法学评分 ≥ 8 分的论文」;Agent 跑了几步、读了几篇之后,可能会按它「认为合理但研究者没要求」的标准重新分类,例如把「方法学评分」隐式重新定义为「样本量」,或者把「筛出」重新解读为「按主题分组」。最终成品看起来仍是一份分组的论文清单,但与研究者的实际要求已经不一致。

漂移的工程补救有几种。其一,在 Agent 提示中要求每若干步重述一次初始目标,并对比当前进度与初始目标的距离。其二,在每个 sub-task 完成后强制写入一份「该步与初始目标的对应说明」。其三,设置「目标变更需要显式触发」的硬约束,使 Agent 无法在没有显式重定目标的情况下静默改变行为定义。这些补救都不复杂,但若不被显式设计进去,默认情况下 Agent 都会漂移。

### 循环:无法完成的步骤被反复重试

循环指 Agent 卡在一个步骤上反复尝试,每次失败、每次又改一点,直到 token 烧光或调用方强制中断。其典型机制是:Agent 缺少「主动放弃」的能力,因为它的提示通常没有显式给出「何时应当放弃,失败信号是什么」。Huang 等人对 LLM 自纠错能力的评估 [@huang2024selfcorrect] 指出,模型在反复尝试同一类任务时倾向于在表层做无意义的修改,而非有效识别该类任务在当前模型能力下不可解。

具体场景常见于编程 Agent 调试某段代码:Agent 试图调通一段它并未正确理解的逻辑,每次小改一点(改变量名、调整缩进、增删一个分支),三十次后仍未通过测试,但它没有「这一步本身可能不可解,该 abort 给调用方」的判断能力。研究场景中的对应表现是:Agent 试图从一份格式异常的 PDF 中提取表格,反复调用不同的 OCR 工具,每次都失败但每次都换一个工具继续,直到 token 预算耗尽。

循环与漂移的关系是:漂移是 Agent 走出了正确轨道,循环是 Agent 困在了无法走通的局部。二者常常前后衔接出现:Agent 先漂移到了一个研究者没要求的子任务,然后在那个子任务里循环。

工程补救包括:其一,显式重试上限(单步重试 ≤ 3 次,失败后 abort);其二,「失败信号清单」(若出现某类异常,立即停止并报告);其三,失败后的「降级回退」机制,例如「OCR 失败 ≥ 3 次,改为输出文件名 + 失败说明,继续后续步骤」。这些机制的共同特点是把「何时放弃」从 Agent 的隐式判断变成显式规则。

### 静默截断:声称完成但未完成

静默截断指 Agent 并未完成任务,但报告中描述为完成。其典型机制有二:其一,Agent 在长任务中遇到了它无法明确识别为「失败」的边缘情况(例如上下文窗口接近上限、工具调用超时),它选择输出一段相对中性的「该任务已完成」描述;其二,Agent 训练数据中「友好回应」的比重远高于「显式失败报告」,导致它在不确定时倾向于优先输出令人满意的结论。

本书项目在 ch06 起草中经历的故障是这一类失败的真实案例:chapter-author 在两次连续尝试中只输出了章节开篇即自终止,既没有继续后面的二级节,也没有 git commit,但它的报告中没有显式声明失败,只是用相对模糊的语言描述了「已完成开篇部分」。这一类失败的危险性,在五种失败模式中最高;研究者收到「已完成」的报告时,默认相信任务已经完成,直到他自身去检查最终成品才发现实际产出与报告之间的差距。Ji 等人对自然语言生成中 hallucination 的系统综述 [@ji2023hallucination] 指出,当模型被要求报告自身行为时,其报告与真实行为之间的偏差是一类被严重低估的可靠性风险;agent 自评属于这一风险的高发场景。

工程补救的中心思路是:不允许 Agent 自身声明「完成」,完成态必须由调用方独立验证。验证手段可以是机械的(`wc -l` 检查行数、`git log` 检查 commit、`grep` 检查关键标记),也可以是语义的(独立的 reviewer agent 检查产出是否覆盖了所有声明的目标)。共同特点是「完成的判定权不归生成者所有」。这一原则与 §2 介绍的「独立 reviewer」与「落盘原则」是同一组设计的不同侧面。

### 沉默丢弃:中间数据无声消失

沉默丢弃指 Agent 在某一步遇到无法处理的输入,默默跳过,继续后续步骤。最终成品看起来完整,但其中少了一部分,且这一缺失不被 Agent 显式标注。其与第七章「无声丢弃」(数据清洗中 inner join 默认行为悄悄丢弃匹配不上的观测)同构,但发生在 agent 多步之间而非单步代码内。

具体场景:Agent 整理 50 篇 PDF 的引用关系,遇到 3 篇打开失败,默默跳过,最终输出 47 篇的对比表,且未在表中或日志中标记「3 篇失败」。研究者收到的是一份看似完整的 50 → 表的对照,但其中只覆盖了 47 篇。Nature 编辑部 2023 年关于 AI 工具透明性的立场声明 [@nature2023tools] 指出,任何用 AI 处理科研数据的环节,「未被处理的数据」与「被处理但失败的数据」都必须被显式记录;agent 编排的沉默丢弃直接违反这一要求。

工程补救:其一,任何跳过必须显式记录到 audit log,且 audit log 在最终成品中作为附件呈现;其二,工作流的最后一步必须是「核对总数」(例如「输入 50 篇,输出 X 篇,失败 Y 篇,X+Y 必须等于 50」);其三,失败超过某阈值(例如 ≥ 10%)时,Agent 应当 abort 给调用方而非继续。这些补救把「数据完整性」从 Agent 的隐式承诺变成显式工程不变量。

### 级联错误:每步合理而总体偏离

级联错误指第一步的轻微误差被第二步放大,逐步累积,到第五步已经面目全非。其典型机制是:每一步「相对于上一步」都是合理的,但累计相对于初始目标已经显著偏离,Agent 自身因为只能看到「当前步与上一步」而无法识别这一偏离。

第二章曾经给出过一个朴素的数学示意:若每一步的可靠性是 95%,十步之后整体的端到端可靠性是 0.95¹⁰ ≈ 60%;若是 90%,十步后约为 35%。这一数学是简单的,但它的含义在 Agent 场景下是深刻的:Agent 跑得越长,其端到端可靠性下降得越快,而调用方常常以「单步质量」为基准评估 Agent 整体表现,导致系统性的过度信任。Dziri 等人的实证 [@dziri2024faith] 给出了与此一致的图景:模型在多步组合任务上的失败率随步数显著上升,且这一上升不能通过「让每一步的提示更精细」线性补偿。

具体科研场景:Agent 从一个研究 hypothesis 出发,经过 5 步「重新表述、扩展、聚焦、改写为可检验形式、生成可执行计划」,最终给出的研究计划与研究者的初始 hypothesis 之间已经只剩下表层的关键词重叠;Agent 的每一步都是合理的,但合理性是局部的、相对于上一步的。

工程补救:其一,设置阶段性检查点(每完成若干步,把当前状态与初始目标显式对照);其二,在 audit log 中显式记录「每步与初始目标的距离」(可以是简单的语义距离,也可以是嵌入向量的余弦距离);其三,引入独立的「初始目标守卫」,即一个 sub-agent 专门负责在每一阶段检查现在做的事是否还与初始目标对齐。这些补救都增加了工作流的复杂度,但级联错误的隐蔽性恰好需要这种额外的复杂度来对抗。

::: {.callout-warning}
## 自动化偏见的复合机制

第四章 line 331 提出过自动化偏见,即人类倾向于接受自动化系统的建议,即使这些建议与自己的判断相矛盾 [@risko2016cognitive; @sparrow2011google]。在 Chatbot 模式下,这一偏见每次只针对一次回应起作用,损失被局限在「一次接受了不该接受的建议」。Agent 模式下,偏见的作用对象变成了一连串自动化决策;研究者审视 Agent 的最终成品并觉得「看起来对」时,他同时接受了背后所有他没有看到的中间步骤,而那些步骤可能涉及上述五种失败模式中的任何一种或多种。

这一复合并非加法,而是乘法。若研究者对单步建议的接受意愿是 80%,对 10 步的整体接受意愿可能高达 95% 以上,因为研究者不再有「逐步审视」的认知带宽,他被迫以「整体看起来合理」为唯一判据。Kahneman 对认知偏差的系统综述 [@kahneman2011thinking] 把这一类整体判断刻画为系统 1 的典型失败:系统 1 在面对超过其处理能力的复杂对象时,倾向于以「整体观感」替代「逐步分析」,而 Agent 编排的设计恰好把研究者推入这一失败区。

应对原则可以归结为一句话:Agent 编排的可信度,取决于它的中间步骤是否可被独立审计。具备 audit log、阶段检查点、独立 reviewer 的 Agent,可以在大多数科研场景下被信任;缺乏这些设施的 Agent,无论其单步能力多强,都不应被赋予不可逆的执行权。
:::


## Agent 资产的层级与判定 {#sec-agent-asset-judgment}

进入 Agent 模式之后,研究者会迅速积累出一组「工具上的工具」:好用的 prompt、复用率高的 skill、跨多个项目都需要的 sub-agent 配置、自建的 MCP server。这些资产构成了研究者「与 AI 协作的方式」的物质载体,值得被认真治理。本节给出一个分层框架,并给出「同一能力应当封装在哪一层」的判定法。

### 四层抽象:prompt / skill / tool(MCP) / sub-agent

把 Agent 编排中的可复用资产按封装强度分为四层,各层的特征如下。

prompt 层(文本指令,最轻):一段提示文本,可能包含模板变量。封装强度最低,可被任意复制粘贴,无独立的执行环境,无独立的验证逻辑。研究者笔记本里那段「请帮忙审一段方法论」式的提示,就属于这一层。优点是创建成本极低,缺点是缺乏版本管理、缺乏验证、跨工具迁移困难。

skill 层(封装的指令 + 模板 + 验证逻辑,中等):一组结构化文件(通常是 markdown + 元数据),包含触发条件、输入输出协议、内嵌的硬约束。Claude Code 的 skill 是这一层的典型实现。skill 与 prompt 的差别在于:skill 有显式的协议(它声明自身接受什么输入、输出什么、何时应当被调用),而 prompt 只是一段文本。skill 的版本管理、跨项目复用、与其它 skill 的协作都被显式设计。

tool / MCP server 层(可被 agent 调用的外部能力,较重):一个独立运行的程序,通过 Model Context Protocol(MCP)或类似协议向 agent 暴露能力。tool 与 skill 的差别在于:tool 是在 agent 之外执行的,通常涉及对外部世界的接口(读写文件、查 API、调用数据库、运行子进程),agent 通过协议调用之并接收结果;skill 则在 agent 的同一执行环境内运行,在原理上是一组结构化的提示。

sub-agent 层(独立 context 中的另一个 agent,最重):在主 agent 之外,以独立 context、独立指令体系运行的另一个 agent。sub-agent 与 tool 的差别在于:tool 通常执行确定性任务(给定输入,输出可预测),sub-agent 自身具有 LLM 的非确定性,其执行过程是另一个完整的 agent 生命周期。chapter-reviewer 在本书项目中就是 chapter-author 的 sub-agent。

第七章 @sec-coding-abstraction-levels 已经为编程场景给出过 prompt / skill / tool / MCP 的相邻分层。本节的不同之处在于把 sub-agent 显式作为第四层加入,以适配编排中「需要独立判断视角」的工程需求。

### 判定法:同一能力应放在哪一层

研究者在面对一项 Agent 编排需求时,常常困惑于「这个能力应该写成 prompt 还是 skill,应该是 tool 还是 sub-agent」。下面给出一个由四个维度联合决定的判定法。

第一维度,**复用频率**:这一能力的预期使用频次是多少?一次性的实验性想法属于「单次」级别;每周用一次的属于「常用」级别;每章/每个项目都用属于「频繁」级别;每天都用且跨多个项目属于「极频繁」级别。

第二维度,**验证强度**:这一能力的产出是否需要被独立验证?完全不需要(纯生成任务)属于「无验证」;需要轻量验证(模板格式、字段完整性)属于「模板验证」;需要硬规则验证(例如「BibTeX key 必须存在于 references.bib」)属于「硬规则验证」。

第三维度,**外部接口需求**:这一能力是否需要与外部世界交互?完全不需要(纯文本生成)属于「无接口」;需要读写本地文件属于「文件接口」;需要调用网络 API 属于「API 接口」;需要访问数据库或运行子进程属于「重接口」。

第四维度,**判断独立性**:这一能力是否需要与主 agent 不同的判断视角?完全不需要(只是执行)属于「无独立判断」;需要某种形式的独立审视(质量检查、异常识别)属于「需要独立判断」。

把四维度联合起来,可以得到下面的近似决策树:复用频率为「单次」、验证强度为「无验证」、外部接口为「无接口」、判断独立性为「无独立判断」,采用 prompt;复用频率为「常用」或更高、验证强度为「模板验证」、外部接口为「无接口」或「文件接口」、判断独立性为「无独立判断」,采用 skill;外部接口为「API 接口」或「重接口」,采用 tool / MCP server;判断独立性为「需要独立判断」,采用 sub-agent。

这一判定法不是机械规则,真实场景常常需要权衡。但它把「资产应放在哪一层」从一个隐式的工程直觉变成显式的多维评估,后者比前者更可被讨论、更可被追溯、更可被在团队之间共享。

### 资产版本化:用 git,不是用平台

Agent 资产积累到一定规模后,版本化与协作管理成为不可回避的工程问题。研究者的常见误判是把这些资产留在 AI 平台自身的「保存的 prompt」「Project」「自定义 GPT」一类设施中。这些设施提供了便利的发现机制(例如「之前的 prompt 在何处」)与共享机制(例如「自定义 GPT 共享给同事」),但代价是平台锁定。Bommasani 等人对 foundation models 生态系统的早期分析中已经预警过这一锁定 [@bommasani2022opportunities]:当用户的工作流依赖某一平台特有的设施时,平台变更价格、变更 API、关停产品都将直接造成用户工作流的不可逆损失。

更可靠的做法是把所有 Agent 资产存为本地 markdown 文件 + git 版本管理。这一做法的具体收益:其一,可迁移,任何文本编辑器都可以读,任何 git 服务器都可以托管;其二,可追溯,每一次修改都有 commit message,半年后回看仍能理解变更原因;其三,可分享,可以以仓库形式发布到 GitHub,或以子模块形式被其它项目引用;其四,可回滚,任何修改都可被撤销,与此对照,平台上的 prompt 编辑常常没有版本历史。

具体实现细节上,有两条经验值得明示。其一,每一个 prompt 或 skill 文件的开头都应注明「解决什么问题、什么时候用、什么时候不用」;半年后回头看,这一段元说明的价值常常超过 prompt 本身。其二,重大修改的 commit message 应描述「为什么改」而不是「改了什么」;「prompt: 收紧 finding-citations 拒绝模糊来源」远优于「update prompt」。第八章 @sec-write-first-then-ai 强调过写作即思考,Agent 资产的命名与 commit message 同样如此:写清楚「为什么改」的过程本身,就是研究者对 Agent 设计的再一次反思。

### 跨项目迁移与平台锁定

Agent 资产的另一治理维度是「跨项目迁移性」。研究者常常发现某项目里写的 skill 在另一项目里也需要,但若把它复制粘贴过去,后续的更新就会出现两个不一致版本。Claude Code 的「用户级 skill 目录」(`~/.claude/skills/`)与「项目级 skill 目录」(`<repo>/.claude/skills/`)的双层设计是这一治理需求的一种工程回应:跨项目复用的资产放用户级,项目特有的放项目级,二者通过同一套加载机制被 agent 消费。

更一般的层面上,MCP 协议被设计为跨 agent 平台,在原理上一个写得好的 MCP server 可以同时被 Claude Code、Cursor、其它 MCP 兼容的 agent 调用。但 2026 年初的现实是:不同平台对 MCP 的支持成熟度差异较大,某些高级特征(资源订阅、动态权限、流式响应)在某些平台上不被完整实现,跨平台移植的实际成本仍然不低。Liang 等人在 HELM 评测框架中对 LLM 跨平台行为差异的实证 [@liang2023helm] 给出了一个相邻图景:同一类任务在不同平台、不同模型上的表现差异显著大于一般直觉所预期,这一差异在 agent 编排场景下被进一步放大,因为 agent 行为是模型行为在多步上的复合。

工程上的可操作建议是:写 skill 与 tool 时优先使用 markdown + 通用文本接口,避免依赖任何平台特有的语法;优先使用 MCP 协议而非平台专有协议;在 README 中明示「该资产已在 X 平台上验证过,在 Y 平台上未验证」;对资产做版本化时,尽量保持向后兼容,而非每次大改打破接口。这些原则都很常规,但在 Agent 资产的早期采用阶段最容易被忽视;一旦资产规模超过几十个,锁定造成的迁移成本将远超早期遵守这些原则的成本。


## 边界条件:Agent 化不是答案的几类场景

前四节论证了什么是 Agent、如何编排 Agent、Agent 怎么坏、如何治理 Agent 资产。本节讨论一组对偶问题:有哪些场景下,研究者应当主动选择不 Agent 化?这一节并非对前面论证的撤回,而是对其边界条件的明示。Agent 化是一种工具选择,而非一种价值取向;把它误用在不适合的场景下,损失常常远超收益。

### 学习曲线期:Agent 化会跳过经验内化

任何工作流的初次掌握过程,本身具有教育价值。研究者通过亲自走完一遍流程的每一步,内化了「这一步为什么这样做、若不这样做会出什么问题、不同情境下应当如何调整」的判断力;这一判断力无法通过观察 Agent 的成品获得,它只能通过亲历过程获得。Bjork 关于「desirable difficulty」的研究传统 [@bjork1994memory] 与 Slamecka 与 Graf 关于「generation effect」的早期实证 [@slamecka1978generation] 给出过一致的结论:学习者在不被外界帮助的情况下经历过的认知挣扎,会在记忆与判断力上留下远比被动接受信息更深的印记。

具体场景:一个刚入门的博士生把整个文献综述交给一个 deep research agent,产出可能在格式与覆盖面上都可用,但研究者对「这个文献群里哪些工作是奠基性的、哪些是外围的、哪些之间存在隐性的争论」没有建立起任何独立判断。等到他要在自己的论文中评估某一相邻文献的位置时,他无法做出有信息含量的判断,只能重新回去问 agent;而 agent 此时会在它训练数据的覆盖边界内给出一个「合理但平均化」的回应,损失了「研究者本应在文献综述阶段建立的领域感」。

第四章 §认知外包 line 331 引用过 Sparrow 等人关于互联网时代「Google 效应」的实证 [@sparrow2011google]:人们记住「信息存在于哪里」而非「信息本身」,在搜索引擎时代是一个温和的认知重组,在 Agent 时代被推向一个更极端的版本,即研究者甚至记不住「自身用 Agent 做过什么」,因为整个过程被外包给了 Agent。Carr 在《浅薄》中对类似现象的更普及性讨论 [@carr2010shallows] 把这一损失概括为「思考方式的浅化」。Agent 化在学习曲线期对这一浅化机制有直接放大效应。

工程上的可操作判定:在研究者尚未完整走过某工作流的每一步至少 1-2 次之前,不应把该工作流 Agent 化。Agent 化的恰当时机是「研究者已经能在不依赖 Agent 的情况下完成该工作流,但希望节省其中重复性高、判断含量低的部分」;此时 Agent 化是工具优化,而非能力替代。

### 中间步骤即价值的任务

某些任务的科学价值不在产出,而在过程。把这一类任务 Agent 化,等于把任务的中心价值丢弃,只保留产出的形式。

最直接的例子是审稿。审稿这一活动对学术共同体的意义,不在于审稿人最终给出的「Accept / Reject」结论,而在于审稿人逐字读完一篇论文、与论文中的论证过程对话、识别出潜在问题并表述为审稿意见的整个过程。让 Agent 替代审稿,即使产出可读的审稿意见,也已经把审稿的科学功能掏空了:审稿人没有获得领域感的累积,论文作者没有获得有实质内容的同行视角的反馈,学术共同体的质量门没有被一个具有判断力的人类节点检验。第十五章 §同行评议系统性崩盘将沿这一论证走得更远;此处仅指出 Agent 化在审稿场景下是禁忌。

类似的还有论文阅读。让 Agent 读完一篇论文并给出摘要,产出常常可用,但损失的是研究者「与论文对话」的过程:那些「这一段难以理解、停下来想想」「这个图看了三遍才看明白」「这个结论与之前读过的另一篇论文似乎冲突」的微小认知事件,共同构成了研究者对一个领域的内化理解。Flower 与 Hayes 在 1981 年关于「写作即思考」的认知过程模型 [@flower1981cognitive] 提出过一个相对应的论点:写作的科学价值不在文本产出,而在写作过程中思维被强迫结构化的那一过程;阅读对应的是「输入端的对应过程」,Agent 化输入端等价于把这一过程外包。

更普遍的判断准则可以这样表达:若一项任务的产出可以被多种方式实现,但只有「研究者亲自经历」这一种方式能产生附加的认知效益,那么这项任务不适合 Agent 化。研究者可以用这一准则反思自身的工作流:哪些任务的产出对自身重要,哪些任务的过程对自身重要,前者可以 Agent 化,后者不可以。

### 高失败成本与目标模糊场景

Agent 化的收益是「研究者节省的时间」,代价是「Agent 失败时的修复成本」。当后者远高于前者时,Agent 化在工程上不划算。

高失败成本场景的典型代表:给一个学生写推荐信、回复一封需要细腻判断的邮件、做一个高 stakes 的政治性决策邮件、起草一份需要在合作者之间协调的研究伦理声明。这些场景下,Agent 化能省的时间通常在 10-30 分钟范围内,但若 Agent 出错(例如推荐信中包含一个对学生不准确的评价、邮件中隐含了不恰当的承诺),修复成本可能是数月的关系修复或不可逆的声誉损失。Weidinger 等人对 LLM 风险的系统分类 [@weidinger2022taxonomy] 中关于「representational harms」与「relational harms」的两类风险,在这些场景下被同时激活。

目标模糊场景:研究者自身尚未想清楚要什么,把模糊目标交给 Agent;Agent 跑出来的产物大概率不是研究者实际想要的,但更糟的是,Agent 的产物常常具有「看起来已经把事情做完」的形式说服力,使研究者放弃了「先把目标想清楚」这一本应优先的认知工作。这一类场景下,问题不在于 Agent 不够强,而在于 Agent 化绕过了研究者本应做的判断工作;Agent 化的恰当前置条件是「目标已经清晰,且可以被某种独立方式审计」。

第十二章关于 Supervisor Level 的讨论将沿这一论证给出更精细的边界:AI 在「逼研究者把目标想清楚」与「替研究者把目标做完」之间存在一个微妙的分界,前者是 Supervisor 的合理用法,后者是 Agent 化的误用。本节仅指出:在目标尚未明确之前,研究者应当与 AI 进行 Chatbot 模式的对话(让 AI 帮助澄清目标),而非进入 Agent 模式(让 AI 直接执行模糊目标)。

## 从工具的编排到角色的编排

走到本章末尾,有必要把全章的论证收回到一个更长的弧上。本章的主命题可以这样再陈述:Agent 化是从「单一工具」过渡到「工具的编排」的工程范式跃迁,这一跃迁带来了控制权与可观察性的重大让渡,同时引入了五类典型失败模式与一组新的资产治理需求;研究者要在这一范式下保留对工作流的实际掌控,必须在概念边界、失败模式分类、资产层级判定三组问题上分别建立显式的工程对应。

这一命题在全书脉络中的位置是双向的。向后看,它是第一章 §Agent 是行动者(line 130-138)的具体技术展开,五级框架在 line 122 把 Agent 列为第三级,本章把这一级在工程层面展开为可操作的设计原则;它是第四章 §自动化偏见(line 331)在多步执行场景下的复合放大版,单步偏见在 Agent 模式下被乘以步数,要求警惕程度同步升级;它是第七章 @sec-ai-coding-roles 提出的「补全员/对话员/Agent」三角色分类沿自主程度轴的进一步外推,把一个编程子领域的分类扩展到任意工作流。向前看,本章是第十六章「一个完整的 AI 科研工作流」的工程基础,后者要把研究项目的五个阶段(酝酿、构建、执行、写作、传播)各自的工作流编排起来,本章的设计原则将在那里被全栈应用;它是第十一至第十五章五级研究者角色框架的工程脚手架,RA、Supervisor、Domain Expert、Agent、Governance 五级角色的每一级,都要回答「Agent 化在这一角色上如何具体落地、边界在何处」。

本章定位上是第二部分「AI 工具与技巧」的压轴。从第五章的智能搜索、第六章的文献阅读、第七章的编程与数据分析、第八章的写作辅助,直到本章的工作流编排,第二部分构建的是一个完整的工具栈。但工具栈本身只是手段:研究者不会因为掌握了一组好用的工具而成为更好的研究者,他成为更好的研究者的方式,是把这些工具组织成对应于不同研究角色的工作模式。AI 当研究助理是一种工作模式,AI 当导师是另一种,AI 当领域专家、当独立行动者、当治理对象,各有完全不同的方法论考量。第三部分将沿这一思路展开,而本章是这一展开的工程前提。

本章未解决的问题与开放的边界值得在收束处显式承认。其一,五类失败模式的分类是否穷尽?漂移、循环、静默截断、沉默丢弃、级联错误这一组分类来自当前工程经验的归纳,但 Agent 系统的复杂度仍在快速演化,可能会出现现有分类无法覆盖的新失败模式,例如多 agent 协作场景下的「彼此误信任」、agent 与人类协作场景下的「角色越界」。这些新模式何时浮现、以何种形式浮现,目前没有可靠预测。其二,「prompt / skill / tool / sub-agent」的四层判定法在更大规模工程下是否还成立?本章给出的判定法基于研究者个人或小团队的 Agent 编排经验,在数十甚至数百个 Agent 协同的大规模系统(例如某些领先实验室的自主研究流水线)中,可能需要新的层级与新的判定维度;这部分工程经验目前在公开文献中尚不充分,仅有 Bommasani 等人的 foundation models 系统综述 [@bommasani2022opportunities] 与若干工业实验室的内部报告 [@TODO-large-scale-agent-orchestration-reports-2025] 给出了片段式的图景。其三,Agent 化与学术训练之间的张力,本章在边界条件一节给出了「学习曲线期不应 Agent 化」的论断,但学习曲线的精确长度、Agent 化的恰当切入时机、不同学科与不同阶段研究者之间的差异,本章无法给出统一答案。这些都是研究者各自必须根据自身处境校准的判断,本章只能提供框架而非答案。

最后,本章作为本书项目自身一个 sub-agent 编排(`/draft-chapter`)的产物,它的存在本身就是一个 Agent 化案例。读者在评估本章论证时,可以同时把它作为一个 Agent 编排的 artifact 加以审视:它在哪些段落显得自然,在哪些段落显得机械,在哪些段落体现了对锁包约束的真实回应,在哪些段落只是机械执行了表面规则。这一双重读法本身是 Agent 时代研究者必须发展的一项新阅读能力,它将在第十五章 §论文三种功能(总结/交流/发现)与第十七章 §为产出负全责中以更普遍的方式被讨论。

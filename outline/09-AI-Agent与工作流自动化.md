# AI Agent 与工作流自动化

第二部分到这里,我们讨论的所有 AI 用法都有一个共同点:**你问一句,AI 答一段**。哪怕是上一章「让 AI 帮你润色一整个章节」,本质上仍然是单次的请求-响应。这一章要讨论一个完全不同的模式——**你给 AI 一个目标,它自己拆解、自己规划、自己跨多个工具走完整个流程,把成品交回给你**。

第一章已经埋了这个区分:OpenAI 的五级框架里,第一级是 Chatbot(回应者),第三级是 Agent(行动者)。本书前八章一直在 Chatbot 模式打转,这一章要正面进入 Agent 模式。这不是一个技术上的小升级,而是一个工作方式的根本转变——**你从「每一步都告诉 AI 做什么」变成「告诉 AI 做什么,让它自己想怎么做」**。

随之而来的是一组新问题:agent 失败的方式比 chatbot 复杂得多;你对 agent 的信任校准也要重新建立;你的 prompt 资产、skill 资产、tool 接口需要管理;最关键的是,**当 AI 自己跨多步执行时,第二章「锯齿状的智能」和第四章「自动化偏见」的两条警告会被同时放大**。这一章是 part 2 的压轴,也是 part 3「角色与级别」的技术铺垫。


## 重温:Chatbot vs Agent

第一章给过这个区分,这里值得再梳理一遍——因为日常使用中很多人用 Agent 工具但仍在 Chatbot 思维里思考,反之亦然。

**Chatbot 是回应者。** 你给问题,它给答案。它的任务范围限于「生成下一段文本」这一个动作。哪怕它的回答再精彩,它做的事情本质上是单步的。ChatGPT 的标准对话窗口、Claude 的网页版本、你直接调用 API 的单次请求——都是 Chatbot 模式。

**Agent 是行动者。** 你给目标,它自己拆解。它能调用工具、读写文件、跑命令、查数据库、再根据中间结果调整下一步。你说「帮我从这 50 篇 PDF 里筛选出符合纳入标准的 30 篇,生成一个对比表」,一个真正的 agent 能自己:解析 PDF → 提取关键信息 → 按标准评分 → 排序 → 生成表格。每一步它自己决定怎么做,你不需要在中间介入。

界限不在「用了什么工具」而在「自主程度」。**ChatGPT 联网搜索一下、跑一段 Python 代码——这些已经超越了纯 Chatbot,但它们是「单步外部调用」**;真正的 agent 是「多步、跨工具、有规划、能自主纠错」。

2026 年初的主流 agent 工具大致有几类:**IDE 集成的编程 agent**(Cursor 的 Composer、Claude Code、Continue);**通用对话型 agent**(ChatGPT 的 GPT 体系、Claude 的 Computer Use);**专门的研究 agent**(Elicit 的 Notebook、专业 deep-research 类工具);**用户自建的 workflow agent**(基于 LangChain、LangGraph、或 Claude Code skill/MCP 体系搭建)。本章不打算逐一介绍——和本书一贯的风格一样,工具会换,框架不变。


## 一个真实 agent 工作流的拆解:本书的 `/draft-chapter`

抽象的 agent 概念听起来玄,看一个具体的会清楚很多。

这本书在 `.claude/` 目录下定义了一套自己的 agent 工作流。当作者写完一章的 ideas 文件之后,运行 `/draft-chapter <章节编号>`,会触发以下流程:

1. **前置检查**:确认 ideas 文件存在、`VOICE.md` 存在、章节标题在 `_quarto.yml` 注册过、git 工作树干净。
2. **切分支**:`git checkout -b draft/chXX`。
3. **调用 chapter-author agent**:这个 sub-agent 在独立 context 里运行,读 ideas + VOICE + 已写章节,产出章节草稿到 `outline/XX-*.md`,沿途调用四个 skill(`chapter-template`、`applying-voice`、`finding-citations`、`checking-coherence`),最后 `git commit`。
4. **调用 chapter-reviewer agent**:另一个独立 context 里的 sub-agent,读 draft + VOICE + 其他章节,产出 `REVIEW-chXX.md`,**不修改任何文件**。
5. **Quarto 渲染验证**:跑 `quarto render --to html`,把错误片段附到 REVIEW 末尾。
6. **汇报给作者**:草稿位置、REVIEW 位置、渲染状态、下一步指南。

这个工作流的几个特征值得拆开看,因为它示范了 agent 设计的几条原则。

**特征一:多个独立 sub-agent,而非一个大 agent。** chapter-author 和 chapter-reviewer 在独立 context 里运行,互不可见。这不是冗余——reviewer 不能带 author 的偏见,所以必须独立 context。**当一个工作流里有「质量检查」环节,检查者必须有独立的视角,这往往意味着独立的 agent**。

**特征二:agent 调用 skill,而不是把 skill 写在 agent 提示里。** chapter-author 自己不知道怎么提取 voice、怎么找引用、怎么查重——这些能力被分别封装成 skill。agent 决定什么时候调用哪个 skill,skill 决定具体怎么做。这种分层让单个组件可以独立改进:你可以把 `applying-voice` skill 单独优化,不需要动 chapter-author。

**特征三:严禁编造文献。** `finding-citations` skill 的硬规则之一是「绝不在 references.bib 中添加你无法独立验证存在的条目」。这个约束写在 skill 层而非 agent 层——意味着任何 agent 调用这个 skill 时,这个约束都自动生效。**关键的安全/伦理约束应该放在最底层(skill 或 tool)而不是依赖 agent 的提示词**。

**特征四:agent 只产出可被 git 追踪的文件 + 一行简报回主对话。** 不允许把章节正文贴到主对话回复里。这强制 agent 真的把工作落到磁盘,而不是「在脑子里写完一遍」。

## Agent 的失败模式比 Chatbot 复杂得多

Chatbot 的失败方式简单:它给你的回答错了,你看见,你不用。Agent 的失败方式复杂得多,因为它跑了多步,每步都可能出问题,而且你常常**看不到中间步骤**。

**失败模式一:漂移。** Agent 在执行过程中逐步偏离初始目标。它本来要给你筛 30 篇论文,跑了几步之后开始按某个它认为合理但你没要求的标准重新分类。这种漂移在长 workflow 里几乎不可避免——就像第二章讲的多步推理误差累积一样。

**失败模式二:循环。** Agent 卡在一个步骤上反复尝试,每次失败、每次又改一点,直到 token 烧光或者你强制中断。这种情况常见于「Agent 不知道怎么完成某一步,但也不会主动放弃」。

**失败模式三:静默截断。** Agent 没真正完成任务,但报告说完成了。本书项目本身就经历过这种失败——`chapter-author` agent 在写 ch06 时两次只输出了开篇就提前自终止,没有继续后面几节、也没有 commit,但报告里没有显式声明失败。**这种失败最危险,因为你不知道它失败了**——直到你去检查最终成品才发现。

**失败模式四:沉默丢弃。** 类似于上一章数据清洗的「无声丢弃」,但在 agent 层。Agent 在某一步遇到无法处理的输入,默默跳过,继续后面的步骤。最终结果看起来完整,但实际上少了一部分。

**失败模式五:级联错误。** 第一步的轻微错误传导到第二步、放大到第三步,到第五步已经面目全非。Agent 自己看不出问题,因为每一步「相对于上一步」都是合理的——它没看到「相对于初始目标」的偏离。

## Agent 自动化偏见的复合风险

第四章已经讨论过自动化偏见——人类倾向于接受自动化系统的建议,即使这些建议与自己的判断相矛盾 [@risko2016cognitive]。**Agent 把这个风险复合化**:不只一个建议要被你信任,而是一连串自动化决策要被你信任。

如果你审 agent 的最终输出觉得「看起来对」,你就接受了背后所有中间步骤——但那些步骤你根本没看到。这是 Chatbot 时代不存在的风险。

**应对原则:任何重要 agent 任务,你必须能 audit 中间步骤,不只是看最终输出**。把 agent 配成「跑完每一步打印一次状态」「把所有中间产物写进文件」「每跑 N 步暂停等你确认」——这些设计上的小约束能让你保留判断权。
:::


## 设计可靠 agent 的几个原则

从本书项目的实践 + 上面的失败模式分析,可以提炼几条设计 agent 的实操原则。

**原则一:把目标拆得足够细,小到每一步可以独立验证。** 「写一整章」太大;「写一节,然后立刻 Edit 到 target 文件,然后 wc -l 验证行数增长」就可以——每一步成功与否有明确的可观察信号。Agent 失败时,你能定位到具体哪一步出问题。

**原则二:每一步的产出落到磁盘,不要只在 agent 内存里。** 这是本书 chapter-author agent 修复 commit 的核心教训:磁盘文件 + git log 才是真实产出,agent 回复的文字不算数。「跑完后才统一保存」是常见的失败种子。

**原则三:在 skill / tool 层加硬约束,不要依赖 agent 提示词记住。** Agent 提示词里的「绝不编造文献」很容易在长上下文里被它自己淡忘;skill 层的「拒绝任何无法验证的 BibTeX 输入」是机械的硬执行。

**原则四:重要 agent 必须有 audit log。** 可以是一个文件,里面按时间戳记录 agent 每一步做了什么、调用了什么 skill/tool、得到了什么中间结果。事后回看时,你能追溯整个决策链。

**原则五:agent 失败时的处理要明确。** Agent 卡了应该 abort 报错给上层,不应该「悄悄跳过这一步继续」。前者你能修,后者你不知道。

## 资产管理:prompt、skill、agent 怎么版本化

进入 agent 工作流之后,你会迅速积累一组**资产**:好用的 prompt、复用率高的 skill、跨多个项目都需要的 agent 定义、自建的 MCP server。这些资产值得被认真管理,理由很简单——**它们是你工作方式的一部分,丢了之后很难重建**。

资产管理的几个最低标准:

- **用 git 追踪,不要存在云端的 webpage 里。** 大多数 AI 平台的「保存的 prompt」「自定义 GPT」「Project」功能都是平台锁定的。把 prompt 和 skill 存成本地 markdown 文件、用 git 管理,你就有了可迁移、可回滚、可分享的资产。
- **每个 prompt/skill 上面写清:解决什么问题、什么时候用、什么时候不用。** 半年后回头看,你会感谢这个习惯。
- **跨项目复用的部分提到一个公共目录(像 `~/.claude/skills/`),项目特定的部分留在项目里(像 `<repo>/.claude/skills/`)。** Claude Code 的双层 skill 设计就是为了支持这种分层。
- **重大更改写 commit message 解释为什么。** 「prompt: 收紧 finding-citations 拒绝模糊来源」比「update prompt」有用一万倍。

## Agent 何时不该用

最后给一份保留场景:**这些场景下,坚持用 Chatbot 而非 Agent**。

**第一,你正在学一项新工作流时。** 直接上 agent,你学到的是「这个 agent 帮我做了什么」,不是「这个工作流本身是怎么运作的」。先用 chatbot 模式手动走一遍流程,理解每一步的逻辑,然后再考虑哪些步骤值得 agent 化。

**第二,任务的中间步骤你必须看到时。** 比如审稿——审稿的价值在于你逐字读完一篇论文的过程,不在于最终给出意见。让 agent「替我审完这篇论文」失去了审稿的全部意义。

**第三,失败成本远高于自动化收益时。** 给一个学生写推荐信、回复一封重要邮件、做一个高 stakes 的决策邮件——这些场景下,「让 agent 自动跑」可能省 10 分钟,但出错一次的代价可能是几个月的修复。

**第四,目标本身模糊时。** Agent 适合「目标清晰但执行繁琐」的任务。如果你连「我到底要什么」都没想清楚,agent 跑出来的东西大概率不是你要的——你不是需要 agent,你需要先想清楚目标。


## 工具到角色:进入 part 3

让我们回顾本章的几个原则。

1. **Chatbot vs Agent 的区别在「自主程度」,不在「用了什么工具」。** 知道你在用哪种模式,才能匹配正确的警惕程度。
2. **本书的 `/draft-chapter` 是一个真实 agent 工作流案例。** 它示范了「多个独立 sub-agent + skill 封装能力 + 硬约束在底层 + 严格落盘」的设计原则。
3. **Agent 失败模式比 Chatbot 复杂得多:漂移、循环、静默截断、沉默丢弃、级联错误。** 这些失败你常常看不到中间步骤,所以更危险。
4. **设计可靠 agent 的几条原则:目标细拆、每步落盘、硬约束在底层、audit log、失败要 abort。**
5. **资产管理:用 git,不要锁在平台里;跨项目复用的提取到公共目录。**
6. **四类场景应该坚持 Chatbot 而非 Agent:学新工作流、要看中间步骤、失败成本高、目标模糊。**

到这里,Part 2「AI 工具与技巧」就走完了。我们从智能搜索(ch05)、文献阅读(ch06)、编程与数据分析(ch07)、辅助写作(ch08),一路到 agent 工作流(ch09)和 ch10 的多模态(下一章)——构建了一个完整的工具栈。

但工具只是手段。真正决定你研究质量的,是**你怎么把这些工具组织成对应不同研究角色的工作模式**——AI 当研究助理是一种用法,AI 当导师是另一种,AI 当领域专家、当独立行动者、当治理对象,各有完全不同的考量。这正是 part 3 要展开的。在进入 part 3 之前,下一章「图像生成与多模态」要补完工具栈的最后一块——文本之外,AI 能处理图、表、音、视频的部分,以及它们在研究中怎么用、怎么不被用错。
